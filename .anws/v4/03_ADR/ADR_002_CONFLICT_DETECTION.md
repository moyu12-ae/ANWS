# ADR-002: 冲突检测与托管文件策略

**状态**: Accepted
**日期**: 2026-03-14
**关联 PRD**: [REQ-004]
**关联版本**: .anws v4

---

## 背景

v4 引入多工具适配后，`anws` 不再只面对单一 `.agents/` 目录。冲突检测的核心问题升级为：

1. 不同目标工具目录下，哪些文件仍然属于 `anws` managed files？
2. 多个目标工具可能共存时，如何避免误伤用户文件？
3. 适配层是否会放大 managed files 边界的不确定性？

---

## 决策

继续坚持 **静态 managed files 清单 + 目标工具作用域隔离 + 单次确认** 策略。

### 核心原则

- managed files 必须基于 `anws` 已知目标路径显式定义
- 目标工具之间的 managed files 作用域独立计算
- 任何自动化更新都不能跨越已解析出的目标工具作用域
- 用户自定义文件默认视为不可触碰资产

### v4 补充规则

- 适配层输出的目标文件路径必须是可枚举、可审计的
- 如果工具目录结构无法被可靠识别，CLI 必须停止并提示人工确认
- 对 legacy 目录的迁移与删除，仍需显式确认

---

## 影响范围

- `manifest.js` 将需要支持多目标工具的路径清单
- `init` / `update` 需要先解析 target tool，再执行冲突检测
- changelog 需要记录目标工具上下文，避免升级分析失真

---

## 候选方案对比

| 方案 | 描述 | 结论 |
|------|------|------|
| 静态清单 + 目标隔离 | 每个工具目标有显式 managed paths | ✅ 采纳 |
| 动态扫描全部目录 | 自动推断所有文件是否可管 | ❌ 风险过高 |
| 哈希数据库 | 通过状态库识别 managed files | ❌ 复杂度过高 |
