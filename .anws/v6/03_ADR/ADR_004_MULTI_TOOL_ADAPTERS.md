# ADR-004: 多 AI IDE 多目标显式安装策略

**状态**: Accepted
**日期**: 2026-03-15
**关联需求**: [REQ-001] [REQ-003] [REQ-004]
**关联版本**: .anws v6

---

## 背景

v5 已明确 `anws` 采用单目标 IDE 安装策略，并据此完成了第一轮 target-aware CLI 落地。但产品需求进一步演进：同一项目需要同时服务多个 AI IDE，因此“单项目单 target”已经从降低复杂度的保护栏，变成限制真实使用场景的边界。与此同时，我们已经确认不能退回到“手动复制多份目录”的无状态模型，否则 `update`、changelog、README 与受管文件边界都会失去权威依据。

---

## 决策

采用 **多目标 IDE 显式安装** 策略：

1. `anws init` 必须允许用户显式选择一个或多个目标 AI IDE。
2. CLI 按 target 分组安装所需目录与文件，物理落盘彼此独立。
3. `anws update` 默认扫描当前项目中的已安装 targets，展示扫描结果后统一更新全部命中 targets。
4. 多目标 `update` 允许部分成功，但必须逐 target 报告结果，并维护安装状态一致性。
5. target 的增删改必须通过显式状态管理实现，而不是依赖隐式目录猜测。

---

## 第一批目标

- Windsurf
- Antigravity / `.agents`
- Cursor
- Claude
- GitHub Copilot
- Codex

---

## 选择理由

- 满足同一项目服务多个 AI IDE 的真实使用场景。
- 保持“共享源 + 独立落盘”的清晰边界，而不是让用户手动复制目录。
- 让 `update` 能围绕已安装 target 集合统一编排，而不是只处理一个上下文。
- 让 README、CLI help、终端输出与真实多目标行为一致。

---

## 后果

### 正面

- 初始化交互能够覆盖真实多目标场景
- manifest 可按 target 集合动态生成
- update 可基于项目中全部已安装 targets 统一执行
- 新增目标 IDE 时仍只需补充一套 adapter 规则

### 代价

- 需要增加多目标选择交互与状态管理
- 需要定义 install-lock、目录扫描兜底与漂移修复逻辑
- 需要定义部分成功、target 移除与重建状态等生命周期语义

---

## 约束

1. 多目标安装必须是显式选择结果，不能隐式推断并自动扩装
2. 目标差异不得直接散落在 `init.js` / `update.js` 的复制逻辑中
3. 终端输出必须展示当前命中的 target 集合上下文
4. 多目标安装不意味着共享物理文件；每个 target 的落盘结果必须独立
